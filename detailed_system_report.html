1. Executive Summary

The SPS Enterprise AI Proposal Capture Manager (SPS-RFP) is a production-grade, stateful, multi-agent platform designed to automate and formalize response processing for Requests for Proposals (RFPs).

Relational Tables 22 Fully Automated migrations
Specialized Writers 15 Segment-mapped personas
Review Agents 14 Departmental check gates
Orchestration Nodes 8 LangGraph state cycle

Traditional proposal response frameworks suffer from high human latency, inconsistent styling, compliance drift, and poor historical knowledge retrieval. The SPS-RFP platform resolves these gaps by integrating structured document ingestion, hierarchical section classification, semantic requirement parsing, automated business-rules checking, opportunity win estimation, outline work-breakdown generation, and a swappable hybrid Retrieval-Augmented Generation (RAG) core.

Core Design Principles
  • Zero-Hallucination Fallbacks: Mandatory citations matching source documents, appending explicit content gap markers if context is missing.
  • Human-In-The-Loop (HITL) Controls: Strict audit logs and manual bypasses at four specific lifecycle approval gates.
  • Decoupled Architecture: Standardized agent registers and model execution layers that prevent tight vendor lock-in.

4.1 Swappable Registries

The system isolates prompt configurations, capability boundaries, and LLM endpoints into dedicated database registers:

  • Model Registry: Declares API endpoints, fallback models, temperature settings, and cost metrics for Gemini 2.5 Flash, Claude 3.5 Sonnet, GPT-4o, and local Ollama integrations.
  • Prompt Registry: Version-controlled repository of system instructions. Prompts are seeded in database tables to isolate engineering updates from runtime builds.
  • Capabilities Registry: Defines what tools and database permissions each agent class inherits during runs.
  • Workflow Registry: Registers stateful LangGraph configurations mapping target node classes to state fields.

6. End-to-End Enterprise Workflows

The platform features modular execution pipelines, running background operations via Celery workers to maintain frontend responsiveness.

6.1 Document Ingestion & Classifications (Phases 3 & 4)

  1. Physical File Ingest: The client uploads a PDF/Word file to /api/v1/documents/upload. The LocalStorageService writes the binary to the local /storage/ volume and commits a database record in the rfp_document table.
  2. Structural Extraction: The DocumentProcessorFactory initiates a processor based on file mime type (e.g. PDFProcessor via PyMuPDF or DOCXProcessor via python-docx).
  3. Hierarchical Parsing: The parser loops through paragraphs, identifying headings, parent titles, tables, page offsets, and structures. It passes raw segments to the Gemini 2.5 Flash API to classify headers into standard labels: Introduction, Background, Requirements, Deliverables, Commercial, Legal, Technical, Submission Instructions, or Appendix.
  4. Persistence: The structure is mapped to the document_section and rfp_metadata tables, enabling targeted content lookups.

6.2 Requirement Intelligence & RAG Core (Phases 5 & 9)

Extracted segments undergo detailed analysis to isolate explicit business requirements, compliance obligations, operational assumptions, and delivery risks:

  • Requirement Engine: Evaluates parsed text blocks to identify explicit customer statements (e.g., "Must deliver within 30 days"), assigning a risk score from 1.0 to 10.0. These entries populate interactive SVG dependency graphs in the UI.
  • Semantic Chunking: Knowledge assets are chunked dynamically by headings and indexed via pgvector or local FAISS indexes.
  • Hybrid context retrieval: When a search is triggered, the system combines vector distance searches (70%) with traditional BM25 keyword matching (30%), rerankers filter and score references, and returns citation links mapping the exact source asset.

7. Decision Engine & HITL Gates

The platform enforces business logic through automatic evaluation rules and human approval gates. It prevents proposal generation on unviable opportunities while maintaining audits for overrides.

7.1 The Opportunity Scoring Engine (Phase 7)

The system evaluates bids against configured rulesets to calculate a composite Opportunity Score (0 - 100). Weights can be configured directly from the admin panel:

  • Strategic Fit (20%): Aligning the proposal with corporate business objectives.
  • Capability Alignment (25%): Evaluates if the operations team matches requested capabilities.
  • Financial Viability (20%): Checks project margins and target values.
  • Risk Rating (15%): Deducts score weight based on flagged compliance risks.
  • Compliance Readiness (20%): Checks alignment with mandatory clauses.
Strict Veto Rules
If any department reviews yield a strict NO_GO decision (such as payment terms exceeding 120 days or insurance liability requirements exceeding $10M), the Opportunity Score is automatically capped at 30, and the proposal status is set to ESCALATE. This blocks automated progression to the outline planning phase until an executive override is signed off.

7.2 Human-In-The-Loop Approval Gates

Four key checkpoints require explicit sign-offs, which are stored in the audit trail:

Approval Gate Required Actor Role Required Verification Step System API Enforcement Action
Gate 1: Ingestion & Requirements Proposal Analyst Verify parsed RFP questions, document formatting limits, and meta items. Locks raw requirements tables from edit changes on approval.
Gate 2: Qualification Go/No-Go Business Unit Director Approve the opportunity score, win probability estimate, and department escalations. Blocks planning workspace creation if gate evaluation is incomplete.
Gate 3: Outline & Planning WBS Proposal Manager Lock outline section tasks, milestones calendar, and resource budget values. Freezes plan structure; blocks content generators from execution.
Gate 4: Proposal Assembly Sign-Off Director Review overall compliance checklists, draft reviews, and quality metrics. Generates final Word DOCX bundle and locks project modifications.

8. Platform Security, Governance & Operations

The SPS-RFP platform is built for enterprise deployments, featuring robust authentication, rate-limiting, and content safety filtering layers.

8.1 Security & Access Controls (Phase 14)

  • JWT Token Management: Dual-token authentication system using ephemeral access tokens (15-minute expiry) and persistent refresh tokens (7-day expiry) stored in HTTP-only cookies.
  • Role-Based Claims (RBAC): Endpoints check JWT claims for role assignments: Admin, Writer, or Reviewer. Only admins can access model registry configuration panels and database audits.
  • API Rate Limiting (SlowAPI): Implements token-bucket rate limiters on public endpoints (e.g., 5 uploads per minute, 60 search queries per minute) to protect backend services from denial of service.

8.2 AI Governance Service (Phase 10.5)

Every model transaction passes through the AIGovernanceService block before execution and return:

  • Prompt Injection Detection: Analyzes raw prompts for adversarial overrides, system prompts requests, or instruction jailbreaks. Flags reject codes directly to the agent event bus.
  • PII Redaction: Post-processing scans output payloads to redact emails, phone numbers, and structural identification numbers before they are saved to the database.
  • Content Safety Audit: Scans outputs against corporate blocklists to prevent the generation of unauthorized liability or pricing commitments.

9. Verification & Quality Guardrails

The platform implements strict verification checks and quality scoring metrics on generated sections to prevent AI hallucinations and compliance drift.

9.1 Zero-Hallucination Guardrails

When generating a section, the system queries the hybrid search retriever for validated reference documents. If the retrieved context scores below the required similarity threshold, the generator implements two defense rules:

Strict Citation Rules
  1. Content Gap Indicators: If references are insufficient to answer a section, the system writes a placeholder: [CONTENT_GAP: <reasons>], highlighting the exact information required from bid management.
  2. Mandatory Citation Mapping: Every generated paragraph must map to a database record in the proposal_evidence_link table, identifying the source document name, section path, and match score.

9.2 Automated Quality Scoring

Every generated draft section is processed through a multi-dimensional validator that calculates a quality score from 0.0 to 1.0:

  • Duplicate paragraph detection: Verifies if the model has repeated sentences, reducing scores if duplicate blocks exceed 10%.
  • Word Count compliance: Validates outputs against targets defined in the RFP document structure.
  • Tone and Style scoring: Calculates alignment with the configured target persona ruleset (Professional, Technical, Persuasive).

Known Limitations & Next Steps

  • Docx Layout Styling: Current exporter parses headings and tables accurately, but complex corporate visual designs must be configured manually inside the Word file post-export.
  • Vector Index Latency: In-memory FAISS indices scale efficiently up to 10,000 files, but pgvector should be deployed for larger, enterprise-grade vector storage databases.
  • Dynamic Fine-Tuning: Prompt registries are fully versioned in the database, but future releases will implement feedback loops that run automated downstream model updates on approved sections.
